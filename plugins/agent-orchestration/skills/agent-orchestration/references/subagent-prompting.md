# 子代理 prompt：怎么写一个边界清晰的派发

子代理只拥有你给它的 prompt（加上它自己去读的文件）。prompt 写好了，它聚焦且安全；写糊了，它会自由发挥——多做没要求的事、烧上下文乱读、甚至回滚别人的改动。本文给出通用骨架、按角色的差异化约束、和常见翻车点。

## 通用五件套

每次派发，至少讲清这五件事：

1. **范围**：你只负责什么，到哪为止。（"只实现 X 单元；不碰 Y、Z。"）
2. **必读文件路径**：把它要读的文件路径列出来——规格、相关 handoff、要改的代码。它不会自动读，也读不到聊天记录。
3. **必产出**：要写 / 改哪些文件、要写哪个 handoff、末条消息要回传什么。
4. **禁止做**：明确的禁区（见下"按角色"）。
5. **别覆盖他人改动**：提醒它"你不是唯一操作者，不要回滚或重写不在你范围内的文件"。

再加一条**末条回传格式**，让它结束时列出：改了哪些文件 / 写了哪个 handoff / 跑了什么命令 / 结果 / 未解决项。你靠这条回传来审核与登记台账。

## 一个可套用的骨架

```text
你是【角色名】子代理，参与一个由中枢调度的项目，你不是唯一操作者。

范围：__（只做什么，不做什么）__

先读这些文件（自己去读，我不会把内容贴给你）：
- __/path/to/TASK-SPEC.md__
- __/path/to/handoff/0X_xxx.md__
- __（其它必要路径）__

你要产出 / 更新：
- __（要改的文件 / 要写的 handoff）__

你不能做：
- __（按角色填，如：不写报告、不做最终验收 / 不改代码 / 不修改任何文件）__
- 不要回滚或覆盖不在你范围内的他人改动。

运行 / 测试如需环境信息，见 TASK-SPEC 的环境段；关键命令：__（或直接在此给出）__

完成后在末条消息里列出：
- 修改的文件
- 写入的 handoff 文件
- 运行的命令与结果
- 未解决的问题 / 需要中枢决定的事
```

把空格处按本次单元填实即可。

## 按角色差异化约束

不同角色的禁区不同，写 prompt 时点明，避免越界：

- **只读 / 勘查代理**：禁止修改任何文件，只产出总结 / handoff。*为什么*：勘查的价值是不带副作用地给出事实。
- **实现代理**：只做被指派的实现单元；禁止写报告、禁止做最终验收、禁止顺手实现没要求的功能。*为什么*：越界实现会绕过验收门，也会把范围搅浑。
- **验收代理**：禁止改代码（除非中枢明确授权顺手修）；**必须跑真实命令、在 handoff 里引用真实输出证据**来下结论，不能凭阅读感觉判定通过。*为什么*：验收一旦动手改代码，就既当运动员又当裁判，验收失去独立性；凭感觉的"通过"等于没验收。
- **报告 / 整理代理**：依据证据写，禁止改代码、禁止编造数据。*为什么*：报告的可信度全靠它如实复述证据。

## 模型选择与成本

子代理是完整实例，用更合适的模型省钱省时间：

- 机械、范围清晰、改动少的实现 / 勘查 → 更快更便宜的模型。
- 多文件协调、需要判断的集成 / 调试 → 中档模型。
- 架构设计、复杂验收、全局判断 → 最强模型。

让每个子代理**一次拿全所需**（清晰 prompt + 文件路径），减少"缺上下文 → 反复重派"的来回，这是最大的一笔省钱。

## 常见翻车点

| 症状 | 根因 | 修法 |
| --- | --- | --- |
| 子代理"不知道"前一步结果 | 没给它 handoff 路径，或它没被要求去读 | 在 prompt 里列出必读路径 |
| 子代理多做了没要求的东西 | 范围 / 禁区没写清 | 钉死范围与"禁止做" |
| 子代理改崩了别人的文件 | 没提醒"非唯一操作者" | 加"别覆盖他人改动" |
| 验收"通过"但其实没跑 | 没要求真实命令 + 证据 | 要求引用真实输出 |
| 子代理烧了大量上下文乱读 | 没指定必读文件，让它自己找 | 直接给精确路径 |
| 回传摘要太含糊没法登记台账 | 没给末条回传格式 | 指定回传清单 |
