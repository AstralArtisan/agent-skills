# 流水线设计：怎么为本次任务现场设计阶段

本方法**不预设**任何固定流水线。阶段的数量、名称、顺序，都由你在开工前根据任务现场决定。本文给你一套"积木 + 判断准则"，帮你拼出适合本次任务的那一条，而不是照搬某个模板。

## 把阶段当作可选积木

下面是常见的阶段类型。它们是**备选积木**，不是必经步骤——按任务挑选、增删、改名、合并、拆分。

| 积木 | 干什么 | 什么时候需要 | 什么时候跳过 |
| --- | --- | --- | --- |
| 环境 / 前序勘查 | 确认运行时、工具、已有产物路径，写进台账环境段 | 几乎总要——子代理不能猜环境 | 极简纯文本任务可并入立项 |
| 历史 / 经验学习 | 让只读子代理总结前序成果的结构、风格、约定 | 有"参考前几次 / 现有项目风格"的要求 | 全新、无前序 |
| 需求澄清与规格化 | 把原始需求（文档 / 资料）提炼成 `TASK-SPEC.md` | 需求藏在资料里、或需要二次确认 | 用户已把规格讲清 |
| 设计 | 在动手前定方案、接口、边界（写进 spec / handoff） | 实现有多种合理走法、或要先对齐 | 实现路径显而易见 |
| 实现 | 真正产出交付物的主体 | 总要 | —— |
| 独立验收 | 独立子代理跑真实命令验证某个单元 | **每个可交付单元后都要** | 不跳过（见不变量 5） |
| 集成 | 把多个并行单元合到一起并验证不打架 | 有并行产出需要汇合 | 全串行、单线 |
| 报告 / 交付物整理 | 依据证据写报告 / 文档 / 演示材料 | 交付物包含报告或说明 | 交付物纯代码且无需报告 |
| 最终验收 | 通盘核对全部交付物、证据、提交是否齐 | 收尾前总要 | —— |

这些只是常见项。任务若有别的天然阶段（数据准备、迁移、压测、合规检查……），照样作为积木加进来。

## 切分单元 + 安排验收门

设计流水线的核心动作，是把交付物切成**可交付单元**，并在**每个单元后面**放一个独立验收门。判断一个好的单元边界：

- 它能被**独立验收**——存在一组真实命令 / 检查，能客观判定它"成了没有"。
- 它的产出能被下游通过**文件**接走，而下游不需要知道它内部怎么做的。
- 它的大小适中：小到一个子代理能在自己上下文里干净做完，大到值得单独一个验收门。

如果一个"单元"无法被独立验收，通常说明它切得不对——要么太碎（合并），要么职责不清（重切）。

## 并行还是串行

- **串行**：单元之间有依赖（B 要用 A 的产出），按依赖顺序一个个来。
- **并行**：单元互相独立、不碰同一批文件，可同时派发省时间；但每个并行子代理都要在 prompt 里被告知"别动别人范围内的文件"，并在它们汇合处加一个**集成 + 验收**步骤检查有没有打架。

并行省时间但更费用量、也更易产生冲突；任务不赶时间时，串行往往更省心。

## 阶段数与成本 / 上下文预算

阶段越多、验收越密，质量越稳，但子代理调用也越多、用量消耗越快。在开工前的立项闸，把"分几个阶段、大约拉多少子代理"跟用户摆明（这正是成本闸的意义）。一个实用的平衡：**关键的、容易出错的、下游依赖重的单元，验收门给足**；琐碎且低风险的小事可以并进相邻单元，别为它单独拉一轮代理。

## 何时临时增删阶段

流水线不是定了就不能改。开工后若发现：

- 某个单元反复验收不过 → 可能切得太大，**拆成更小的单元**各自验收。
- 出现了立项时没料到的依赖 → **插入**一个前置勘查 / 设计单元，并在台账 Decisions 记下原因。
- 某个计划中的阶段被证明不必要 → **删掉**，同样在 Decisions 记一笔。

调整流水线属于你的自主范围（开工后不必回来问用户），但每次增删都要**更新台账与原生记忆文件里的流水线概览**，让文档始终等于现实。
